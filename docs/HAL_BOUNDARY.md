# HAL Boundary Guide — How EmberSim Replaces Hardware

When your firmware runs on real hardware, HAL calls like `HAL_UART_Transmit()`
talk to physical registers on an STM32 chip. When it runs under EmberSim, those
same calls talk to software models. This guide explains how that works and how
to set it up correctly.

## The concept

```
Real hardware:                    EmberSim:
                                  ┌─────────────────────┐
  firmware.c                      │  firmware.c          │
      │                           │      │               │
      ▼                           │      ▼               │
  HAL_UART_Transmit()             │  HAL_UART_Transmit() │
      │                           │      │               │
      ▼                           │      ▼               │
  USART2->DR = data               │  mock_uart.c         │
      │                           │      │               │
      ▼                           │      ▼               │
  physical TX pin                 │  trace.jsonl         │
                                  │  "UART TX: OK\n"     │
                                  └─────────────────────┘
```

EmberSim intercepts HAL calls at the function level. Your firmware code doesn't
change — the linker resolves `HAL_UART_Transmit` to EmberSim's implementation
instead of ST's.

## The include replacement

In your firmware source files, replace the hardware HAL include:

```c
// Before (hardware):
#include "stm32f4xx_hal.h"

// After (simulation):
#include "mock_hal.h"
```

`mock_hal.h` is auto-generated by `embersim init` from your HAL header. It
declares every HAL function found in your header, with `__attribute__((weak))`
stubs in `mock_hal.c`. EmberSim's peripheral models (mock_uart.c, mock_tim.c,
etc.) provide strong implementations that override the weak stubs.

### Which headers to include

| Header | Provides | Always needed? |
|--------|----------|---------------|
| `mock_hal.h` | Auto-generated stubs for every HAL function in your header | Yes |
| `mock_tim.h` | `TIM_HandleTypeDef`, timer peripheral model declarations | If using TIM |
| `mock_spi.h` | SPI peripheral model declarations | If using SPI |

Generated mocks also include these (included automatically by the build):

| Header | Purpose |
|--------|---------|
| `trace_log.h` | JSON Lines trace recording |
| `ember_sim_kernel.h` | Simulation kernel API |
| `ember_regs.h` | Register model |

## Entry point model

EmberSim uses a two-function entry model instead of `main()`:

```c
void app_init(void);   // Called once at startup — initialize peripherals here
void app_run(void);    // Called repeatedly in the simulation loop
```

The generated `ember_sim/host_main.c` calls these:

```c
int main(int argc, char **argv) {
    trace_log_init(trace_path);
    kernel_init();
    nvic_enable(28);     // TIM2 IRQ
    nvic_enable(38);     // USART2 IRQ
    app_init();
    kernel_run_until(50000);
    trace_log_close();
    return 0;
}
```

### Adapting from main()

If your firmware uses a standard `main()`:

```c
int main(void) {
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_TIM2_Init();
    MX_USART2_Init();

    while (1) {
        // application loop
    }
}
```

Extract it into the two-function model:

```c
void app_init(void) {
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_TIM2_Init();
    MX_USART2_Init();
}

void app_run(void) {
    // application loop body — called repeatedly by the host runner
}
```

Copy `ember_sim/host_main.c` to your project, customize it if needed, and add it
to `[build.sources]` in `embersim.toml`.

## What the simulation provides

| Peripheral | Model | What it simulates |
|-----------|-------|-----------------|
| TIM | `mock_tim.c` | Counter, period, prescaler, update interrupt, PWM |
| UART | `mock_uart.c` | Blocking TX/RX, interrupt TX/RX, error states |
| SPI | `mock_spi.c` | Blocking transmit/receive |
| I2C | `mock_i2c.c` | Blocking memory read/write |
| GPIO | `mock_state.c` | WritePin, ReadPin, TogglePin with state tracking |
| NVIC | `ember_sim_kernel.c` | Interrupt enable/disable, pending, dispatch |
| Registers | `ember_regs.c` | Read/write with change tracking |

## What is not supported

These patterns don't work with EmberSim and need workarounds:

| Pattern | Issue | Workaround |
|---------|-------|-----------|
| Direct register access (`GPIOA->ODR = 0xFF`) | No hardware registers | Use HAL API (`HAL_GPIO_WritePin`) |
| Inline assembly (`__asm`, `asm volatile`) | No CPU simulation | Replace with C equivalents |
| `__disable_irq()` / `__enable_irq()` | CMSIS intrinsics | Use `embersim init --repair` to generate stubs |
| `NVIC_SetPriority()` without HAL | CMSIS function | Stub is generated automatically |
| `SystemInit()` | Clock configuration | Already handled by host_main.c |

## When a HAL function is missing

If `embersim check` reports a HAL function as MISSING:

```
HAL functions:
  HAL_ADC_Start                         MISSING
    → Add to HAL header or stub manually
```

This means your firmware calls `HAL_ADC_Start()` but it's not declared in the
HAL header you passed to `embersim init`. Options:

1. **Add it to your HAL header** — if the function is declared in a header your
   `-I` paths don't reach, add the right include directory and re-run `embersim init`.

2. **Stub it manually** — add a weak stub in your firmware or a separate file:
   ```c
   HAL_StatusTypeDef __attribute__((weak)) HAL_ADC_Start(ADC_HandleTypeDef *hadc) {
       return HAL_OK;
   }
   ```

3. **Use `--repair`** — `embersim init --repair` generates shims for known CMSIS
   macros. It does not add missing HAL function stubs.

If `embersim check` reports a HAL type as MISSING, the type definition isn't
reachable from your configured include paths. The most common fix is adding
`mock_tim.h` (for `TIM_HandleTypeDef`) or the relevant peripheral mock header.

## The `--repair` flag

```bash
embersim init --repair
```

This analyzes your project and generates compatibility fixes without modifying
your firmware source:

- Creates `ember_sim/mocks/cmsis_shim.h` with stubs for unsupported CMSIS macros
  (`__disable_irq`, `__enable_irq`, `NVIC_SetPriority`, etc.)
- Adds discovered include paths to `embersim.toml`
- Reports HAL type issues that require manual include changes

You still need to add `#include "cmsis_shim.h"` to your firmware sources
where those macros are used.
